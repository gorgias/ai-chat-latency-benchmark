Two structural findings beyond raw latency:
1. Yuma has no front-end chat widget of its own. On every Yuma reference customer checked (Glossier, EvryJewels) the on-site chat is Gorgias Chat — Yuma operates as a back-end automation layer on the helpdesk (Gorgias / Zendesk), not as a consumer-facing chat product. It is not directly comparable to Siena / Sierra / Ada on "chat latency".
2. Ada was deployed but its bot backend returned "experiencing technical difficulties" across both EU and US Loop instances throughout the ~20-min test window — an availability data point in itself.

Methodology & Caveats

How latency was measured. Each widget was driven programmatically on the live customer site. Metric = wall-clock from message sent to the full response being available.

  • Siena (Simple Modern): measured via the chat API — POST /v1/live_chat/messages then poll /v1/live_chat/message until the bot reply appears. Siena delivers the answer atomically.
  • Sierra (Casper): measured at the network layer by intercepting the streaming endpoint sierra.chat/-/api/chat — first byte and stream-close timestamps. This is immune to browser tab throttling.

Caveats. (1) Different customers / product domains (water bottles vs mattresses) — treat as directional, not a controlled head-to-head. (2) Sierra ran on a warm (server-persisted) session with accumulated context, which tends to slow generation — its figures are conservative. (3) Latency varies strongly by query type: simple FAQs (~4.6s on Sierra) are much faster than product-recommendation queries that trigger retrieval (~9.7s). (4) Ada and Yuma could not be latency-tested for the reasons above. A re-test of Ada when its backend recovers is recommended.
